Troubleshooting

Fix it by symptom

Safe, symptom-led checks for the golos v0.3.3 macOS beta. Prefer re-granting permissions and History recovery over destructive cleanup. Never paste API keys, private transcripts, or audio into public reports.

Applies to v0.3.3 Unsigned beta · macOS 13+
i
Quick triage

For intermittent UI or insertion glitches, do not restart first. Use the menu-bar chakra → Export Diagnostics… so a redacted local zip captures the current session; then check menu Permissions for live ✓/✗, focus a real text field, and try Test insertion. Quit only from the menu-bar icon (not only by closing Settings). Reopen Welcome / Setup… anytime for the wizard checks. Export does not fix every failure mode — it preserves evidence for a report.

App will not open (Gatekeeper)

Public v0.3.3 DMGs are an unsigned beta. macOS Gatekeeper may block a normal double-click. Builds are not notarized; signing and a trustworthy update path are roadmap work, not current claims.

  1. Open the DMG for your architecture and drag golos into Applications.
  2. In Finder → Applications, right-click (or Control-click) golos, then choose Open.
  3. Confirm the open prompt. After the first successful launch, normal opens usually work.
  4. Complete onboarding: grant Microphone, Input Monitoring, and Accessibility to golos.app (not only Terminal).
!
Stay safe

Do not disable Gatekeeper system-wide, run elevated quarantine commands from random guides, or delete broad system folders. If the app still will not open, note the exact dialog text and try the matching architecture DMG (Apple Silicon vs Intel) from GitHub Releases.

Source developers: a second start while one instance holds the lock exits with a message that golos is already running. Quit from the menu, or for a terminal-started process use ./dictate.sh quit (or restart). A Ctrl+Z-suspended process still holds the lock.

Hold key does nothing

Default hold-to-talk is fn (globe). You can rebind live in Settings → General to Right Option, Right Command, or F5 (no restart). Left Option / left Command are never treated as the hold key.

Check these first

  1. macOS fn/globe action. Open System Settings → Keyboard and set Press 🌐/fn key to → Do Nothing. Otherwise macOS may steal fn for emoji, dictation, or another system action.
  2. Input Monitoring. Without it, global hotkeys are not seen while other apps are focused. Menu-bar icon → Permissions → grant Input Monitoring to the same identity that is running golos (golos.app or your terminal), then restart golos.
  3. Microphone. Hold may appear idle if capture cannot start. Grant Microphone and retry.
  4. History retry in progress. While a History Retry owns the pipeline, hold/lock may show a short warning (“History retry is still running”) instead of recording. Wait for Retry to finish.
  5. F5 as hold key. When F5 is selected, golos swallows F5 via the event tap so it does not also act as a system key. If the tap is unavailable, behavior is degraded — grant Input Monitoring.
i

Use Welcome / Setup… for the hold-key test pad, or menu Test insertion after permissions look green, to separate “hotkey not seen” from “insert not delivered.”

fn+Space types a space or ends immediately

fn+Space (or your hold key + Space) toggles hands-free locked recording. The Space is swallowed only when the CGEventTap path is active.

  • Space still types into the focused app — Input Monitoring is missing or the event tap could not be created. Golos falls back to observe-only monitors. Grant Input Monitoring to the running identity, restart golos, and try again.
  • Lock ends almost immediately — a very short press can be ignored as an accidental tap (< ~0.3 s of audio is discarded without transcription). Hold a moment longer; for lock mode, press the combo cleanly, speak, then end with a single hold-key press or the combo again. Esc cancels without inserting.
  • Double-tap hold key — only if [hotkey] toggle_combo = "double_fn" in ~/.golos/config.toml (config-only; there is no Settings control). Accidental double-taps can toggle lock unexpectedly; edit or remove that config value if needed.

Source runs log which path is active at startup: combo path: event tap (blocking) vs monitor (observe-only). See Diagnostics.

Recording works but no animation / status words hidden

Idle is intentionally hidden — no bubble at rest. Status UI appears only while recording, locked, processing, or briefly on success.

  • Notch style (default on notched MacBooks): red→orange waveform wings while recording; blue processing strip; green ✓ inserted success hill.
  • Corner style: Settings → General → bubble style corner — a draggable bottom-right pill. Bubble style is applied at startup; change style, save, then restart golos. Machines without a notch fall back to corner even if “notch” is selected.
  • Status words hidden: General includes a control for status text on the animation (show_text). If labels like “processing…” or “✓ inserted” are missing but the waveform still moves, turn status words back on and Save. Live apply covers show_text and sensitivity; style still needs restart.
  • Flat / quiet waveform: raise Input sensitivity in General (display gain only). Also confirm the correct mic is allowed and that you are not muted at the system level.
  • Strip vanishes mid-session (dictation still works): v0.3.3 recreates the notch strip after wake and after 15+ minutes idle (bounded), with improved WindowServer checks. This is recovery, not a promise of perfect visual restoration. Prefer Export Diagnostics… before restarting so the redacted zip keeps visual-state metadata for a report.
  • After sleep / long idle: hold/recording state and the event tap are recovered on wake when possible. If the hotkey stops responding, open menu Permissions, re-check Input Monitoring, then relaunch golos. Export Diagnostics first if visuals disappeared and you plan to report the session.

Processing appears stuck or immediate-repeat fails

After release, the UI enters processing (blue shimmer / elapsed label after a few seconds). Duration depends on network STT, optional formatting LLM, and clip length — there is no fixed “always done in N seconds” guarantee.

  1. Wait for success (✓ inserted or ✓ inserted raw) or a warning notice. Do not force-quit mid-pipeline unless the app is frozen for a long time with no progress.
  2. Esc during processing requests cancel of the pending insert (if insertion has already started, it is too late to undo automatically).
  3. Immediate re-hold while still processing does not start a second concurrent dictation; finish or cancel first.
  4. If a History Retry is running, new recording is blocked until it finishes (busy coordination).
  5. Cloud path: check network, OpenRouter key in Settings → General, and that the selected STT model id is still valid. Local path: see local model.

After success, you can dictate again as soon as the app is idle (success auto-clears after about a second). If every run hangs, open History for error badges and use diagnostics.

Transcription fails (cloud or local)

Failures surface as a short bubble notice and a durable History row when possible. Typical notices include “connect OpenRouter or download local STT”, “speech recognition failed — open History for details”, and “couldn’t hear that — open History to retry”.

Cloud (OpenRouter, default)

  • Set a valid API key in Settings → General (or environment OPENROUTER_API_KEY for source runs). The key is stored in ~/.golos/config.toml (mode 600) — never paste it into issues.
  • Backend should be openrouter unless you intentionally use local MLX.
  • Confirm network access. Provider outages and model availability change over time — switch model or retry later; this page does not promise prices or uptime.
  • Empty transcript on a real clip can be silence, wrong mic, or a provider result. Use History → Show audio if a WAV was retained, then Retry.

Local (MLX, optional)

  • Requires Apple Silicon and the Apple Silicon build (or a source install with the local extra). Intel / cloud-only builds show local model as unavailable.
  • Weights download only after you click Download local (~1.5 GB) — never starts silently. Wait for Local model ✓ ready before selecting the mlx backend.
  • If download fails, check network/disk space and try again later. OpenRouter remains available without the download.
i

Very short taps (< ~0.3 s) are ignored as accidental and do not run STT. Speak a bit longer if you see no History entry at all.

Text was not inserted even though success appeared

The green success state means golos posted insert events (typed keystrokes for single-line text, or clipboard + synthetic Cmd+V for multi-line). It does not mean macOS verified that the target app accepted the text.

  1. Accessibility preflight. Before posting synthetic keys, golos checks Accessibility. If it is missing, insert is aborted (no false green success), History keeps the result, and the bubble warns “Accessibility needed — result saved in History.” Menu → Permissions → Accessibility → grant → relaunch golos.
  2. Focus a normal text field (not a locked PDF, empty desktop, or some secure fields). Compatibility is best-effort for standard macOS text fields — not a universal guarantee. Green success still means events were posted, not that every app accepted them.
  3. Try menu Test insertion with a known-good field focused.
  4. Open Settings → History: if raw/final text is present, use Copy text and paste manually. Retry regenerates text but does not auto-insert into the app.
  5. If the animation or strip misbehaves intermittently, Export Diagnostics… before restarting so the redacted zip retains session state.
  6. Clipboard after multi-line paste. Default restore_clipboard = true (Settings → General) restores the prior pasteboard asynchronously with a changeCount/CAS guard: a user copy made after Golos posts is never overwritten. Green success still means events were posted — not verified text delivery. If a very slow target pastes the restored prior content instead of the transcript, uncheck “Restore clipboard…” (or set [insert] restore_clipboard = false) so the transcript stays on the pasteboard, or use config-only method = "type" for a clipboard-free path.
!
Posted ≠ verified

Partial success label ✓ inserted raw means formatting fell back to the raw transcript, but insert was still posted. Missing text in the field with green success is still a focus/delivery issue in the target app — not “STT failed,” and not proof that Accessibility was missing (that path no longer shows green success).

Recover via History (Copy / Retry / Show audio)

Settings opens on the History tab by default: newest dictations first, with status/error detail, raw and final text when available.

  • Copy text — copies the best available text (final, else raw) to the pasteboard without changing focus. Paste into your app.
  • Retry — starts a new attempt on a background worker (re-run STT when a retained WAV exists; re-format when raw text exists). It appends another History attempt and does not automatically insert into the frontmost app. On success you get “Retry saved — ready to copy.” If live dictation is running, you get “Busy — live dictation running.”
  • Show audio — reveals a retained WAV in Finder when recordings were kept on disk (default on). If recordings were disabled or the file was removed, STT retry is not possible for that run.

Failed pipelines write durable History rows (STT / format / insert stages) so you can recover without re-speaking when audio or text was kept. Original failure lines are preserved; retries add attempts under the same run.

Local model unavailable or download failed

  • Intel Mac or cloud-only install: button shows “Local model unavailable” — expected. Use OpenRouter cloud STT.
  • Apple Silicon app without weights yet: “Download local (~1.5 GB)”. Click once; title becomes “Downloading…”, then “Local model ✓ ready” when the cache has config + weights. Only then select backend mlx and Save.
  • Download failed: stay on OpenRouter, free disk space, check network, retry the download later. Do not delete random cache trees from third-party tips; use the in-app button again.
  • “Fully local” product setup needs Apple Silicon local MLX and formatting off and learning reviewer off. Cloud STT still uploads audio even if formatting is disabled.

Permissions reset / regrant (golos.app vs Terminal)

macOS treats golos.app and your Terminal / iTerm (or other launcher) as separate identities. Grants to one do not apply to the other. Replacing an unsigned DMG build can also create a new TCC identity — you may need to regrant the three permissions even if they looked fine before the update.

Permission Why golos needs it
Microphone Audio capture for dictation
Input Monitoring Global hold key and event tap (swallow Space / F5 when configured). After granting, relaunch golos so the tap can install.
Accessibility Synthetic insert (type / Cmd+V) and reading focused context. Preflighted before insert so a missing grant does not look like success.
  1. Quit golos completely from the menu-bar icon.
  2. Open System Settings → Privacy & Security.
  3. For each of Microphone, Input Monitoring, and Accessibility: find golos (or your terminal if you run from source). If the toggle is on but behavior is broken, turn it off, quit settings, turn it on again, or remove the row and re-add on next launch.
  4. Relaunch the same identity you granted (Applications → golos, or ./dictate.sh). This is especially important after granting Input Monitoring.
  5. Confirm live ✓ in menu Permissions and finish Welcome / Setup… if anything is still ✗.
  6. Also set Keyboard → Press 🌐/fn key to → Do Nothing when using fn as the hold key.
×
Do not

Do not use elevated tools to rewrite TCC databases, and do not delete broad home-library trees to “reset permissions.” That is unsafe and unnecessary for normal regrant.

Export Diagnostics… and local logs

From v0.3.2, golos keeps private rotating logs under ~/.golos/logs/ and offers an explicit menu command Export Diagnostics… (menu-bar chakra icon). Prefer export before restarting when the strip, animation, or insert path misbehaves intermittently.

How to export

  1. Reproduce the issue once if you can (optional but helpful).
  2. Menu-bar chakra → Export Diagnostics….
  3. Choose where to save the zip (default name like golos-diagnostics-YYYYMMDD-HHMMSS.zip).
  4. Inspect the zip yourself. Share it only if you choose — nothing uploads automatically.

What the bundle includes / excludes

  • Includes: sanitized rotating logs; build / system / permission metadata; allow-listed settings (no secrets); recent History run-status metadata (status, stage, error strings, lengths — not transcript text).
  • Excludes: API keys and credentials; transcript and final text; focused / visible context content; custom prompts; audio files. Do not re-add those when filing a public issue.

Other useful surfaces

  • Settings → History — status, stage, error text, raw/final when stored, Copy / Retry / Show audio. Best first stop for a single failed run.
  • Menu Permissions — live ✓/✗ plus deep links into System Settings. Startup also logs missing permissions.
  • Source / developer runs — ./dictate.sh still prints INFO to the terminal (permission warnings, combo path event tap (blocking) vs monitor (observe-only), STT exceptions).
i

Diagnostics help investigate; they do not claim every UI or insertion failure is fixed. Optional formatter debug logging in config can include prompt content — leave it off for normal use, and never share those logs publicly.

How to file a safe bug report

Use GitHub issues on Dopomogai/golos for product bugs. For security-sensitive findings, follow SECURITY.md (private vulnerability reporting) — do not post exploit detail publicly.

Include

  • Version (v0.3.3 or about string / commit if source-built)
  • Architecture (Apple Silicon or Intel) and macOS version
  • Whether you run golos.app or a terminal source launch
  • Hold key setting and Permissions ✓/✗ summary (three toggles)
  • Clear steps to reproduce and what you expected
  • Sanitized History stage/error strings (e.g. “stt returned empty transcript”) without the transcript text
  • Optional: the redacted Export Diagnostics… zip (you inspected it first)

Never include

  • API keys, tokens, or a full config.toml
  • Private dictation text, window titles that expose secrets, or personal names you care about
  • Audio files or links to personal recordings under ~/.golos/recordings/
  • Screenshots of Settings that show a filled-in key field
  • Unredacted formatter-debug logs or prompt dumps

Contributions and PR expectations: CONTRIBUTING.md.
